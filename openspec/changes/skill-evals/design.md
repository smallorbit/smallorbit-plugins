## Context

Skills here are not pure functions; they are runbooks whose output is **side effects** (PRs, labels, git config, worktrees) plus a report. That rules out simple input→output assertion as the primary strategy. Two repo decisions make evals tractable anyway:

1. **Deterministic work is already extracted into `scripts/`** that emit bare-payload JSON on stdout and human-readable errors on stderr (per `plugins/_shared/script-authoring.md`). Structured output is directly assertable.
2. **Six skills already ship `scripts/test.sh`** — a real, if ungated, L1 foundation.

The gap is everything above the scripts: the model's adherence to the runbook, and the structural integrity of the SKILL.md surface itself.

Reference: Anthropic, *Demystifying evals for AI agents* (the canonical pattern set; no first-party harness exists). Three grader types (code-based → model-based → human), judge calibration against human labels, and "read the transcripts" as the core discipline all come from there.

## Goals / Non-Goals

**Goals**
- A regression net that runs on every PR for the deterministic and structural layers (L1, L2), with near-zero cost and no LLM.
- A behavioral eval harness (L3) for high-blast-radius runbook decisions, runnable headlessly and gradeable deterministically where possible.
- A convention so new skills ship evals the way they now ship `test.sh`.
- Every confirmed audit finding leaves behind a regression check.

**Non-Goals**
- A turnkey framework; LLM evals in the blocking path; abstract model-quality measurement; replacing runtime self-checks.

## The four layers in detail

### L1 — Script unit tests
- **Runner**: a top-level `scripts/run-skill-tests.sh` that discovers every `plugins/*/skills/*/scripts/test.sh` and runs it, failing on any non-zero exit. (Bash-loop convention: pipe `find ... | while read`, never `for N in $(...)`.)
- **CI**: a required GitHub Actions job on PRs touching `plugins/**`.
- **Convention**: `test.sh` becomes mandatory for any script-backed skill; add to `script-authoring.md`.
- **Coverage targets**: the deterministic invariants with blast radius — preflight's cross-pin guard, gather's DAG/topo-sort, teardown's conditional unset, clean-worktrees' stuck-detection.

### L2 — Skill-doc lint (no LLM)
A single linter (`scripts/lint-skills.{sh,ts}`) over the `plugins/` tree, each rule a discrete assertion with a file:line error:

| Rule | Catches | Audit tie-in |
|------|---------|--------------|
| Frontmatter present (`name`, `description`) | malformed skills | — |
| `## Input` table present where a skill takes args | undocumented surface | — |
| Every `<!-- include: path -->` resolves | broken includes | — |
| Every `_shared/*.md` citation path resolves | dead cross-refs | #1052 (root-spec claim) |
| Every relative markdown link resolves | moved targets | #1052 |
| README flag-matrix ⟷ SKILL.md `## Input` agree | silent matrix drift | CLAUDE.md note |
| No `develop` branch reference outside migration docs | single-trunk residue | #1049 (+ `deploy-site.yml`) |
| Allowlist entries point at existing scripts | dead permission surface | #1052 |
| Shared specs are cited, not paraphrased (heuristic) | divergent inlined copies | #1047 |

Rules are additive; each new audit finding that is statically checkable becomes a rule here.

### L3 — Behavioral evals (LLM-in-loop)
- **Fixtures**: small seeded git repos under `evals/fixtures/<scenario>/` (committed or generated by a setup script), e.g. `epic-8-children-one-blockedby`, `epic-labeled-no-subissues`, `pin-already-set-foreign-feature`, `single-epic-arg` (the EPIC_MODE ambiguity), `closes-multiref-one-line`.
- **Runner**: invoke the skill headlessly and capture the trajectory.
  - CLI: `claude -p "<invocation>" --output-format stream-json --max-turns N --permission-mode acceptEdits --max-budget-usd X` — parse the stream-json events for the tool-call sequence and final result.
  - SDK (preferred for the maintained suite): Agent SDK `query(prompt, options)` with `PreToolUse`/`PostToolUse` hooks for clean programmatic tool-call capture and `permissionMode: "acceptEdits"`.
- **Graders (dual)**:
  - *Programmatic (preferred)*: assert observable state — "PR opened with base `main`", "`claude.flowkit.prBase` unset after teardown", "topo-order placed #1051 after #1046", "EPIC_MODE resolved to X for a single epic arg".
  - *LLM-as-judge (fuzzy only)*: plan correctness, PR-body conformance to `pr-body.md`. Pin the judge model; tight rubric with explicit pass/fail; **validate against human labels on a sample before trusting it**.
- **Discipline**: one decision per eval; assert on *structured behavior* (tool calls, resulting state), never golden-file the model's prose.

### L4 — End-to-end smoke
- A few full-flow runs against a throwaway remote (local bare repo + `gh` against a sandbox, or a dedicated test org repo).
- Assert end state: issues closed, branches deleted, pin clean, epic closed.
- Nightly or manual `workflow_dispatch`; never blocking per-PR.

## Tooling decisions

- **No bespoke framework.** `claude -p` (stream-json) for scripts/debugging; Agent SDK `query()` for the maintained L3 suite. A thin grader layer (bash assertions + a small judge wrapper) is all the custom code needed.
- **Determinism** (per Anthropic CI guidance):
  - Pin **exact** model IDs in evals (`claude-sonnet-4-6`, not the floating `sonnet` alias).
  - Low/zero temperature; JSON output for reproducibility.
  - Seed fixtures; scripts already avoid wall-clock/random nondeterminism — keep that.
- **Cost control**: L1/L2 free and per-PR; L3 curated and label-gated (or on `plugins/<changed>/` only); L4 nightly. `--max-turns` / `--max-budget-usd` guardrails on every headless run. Haiku for fast iteration, Opus/Sonnet for final validation.
- **Flakiness**: read transcripts when an eval fails — most failures are grader/harness bugs, not skill bugs. Quarantine flaky L3 evals out of the blocking path until stabilized.

## Layout

```
evals/
  fixtures/<scenario>/            # seeded repos / inputs
  l3/<skill>/<decision>.{sh,ts}  # behavioral evals, one decision each
  graders/                       # shared programmatic + judge helpers
  README.md                      # how to run each layer
scripts/
  run-skill-tests.sh             # L1 discovery runner
  lint-skills.sh|ts              # L2 linter
.github/workflows/
  skills-ci.yml                  # L1 + L2 (required, per-PR)
  evals-nightly.yml              # L3 curated + L4 smoke (scheduled / dispatch)
plugins/_shared/
  eval-authoring.md              # new convention: how/when to add evals per layer
```

## Key decisions & trade-offs

- **Why gate only L1/L2 per-PR?** They are deterministic and free; LLM evals are neither. Blocking merges on a flaky/expensive judge erodes trust in the gate.
- **Why fixtures over mocking `gh`/`git`?** Real git/`gh` against a sandbox exercises the actual side effects the skills produce; mocks drift from reality and miss the bugs that matter (e.g. the pin leak).
- **Why "one decision per eval"?** "Does swarm work?" is unfalsifiable and flaky; "does swarm resolve EPIC_MODE correctly for a single epic arg?" is a crisp, stable assertion.
- **Why OpenSpec for this proposal?** It is the repo's native review surface and lets the approved plan dispatch into implementation via `/opsx:apply` or `opsx-bridge` → squad/swarm.

## Resolved decisions

Settled during review (2026-06-02):

1. **EPIC_MODE for a single epic arg → epic mode.** `/swarm <epic#>` where the one argument is an epic that expands to N children SHALL resolve `EPIC_MODE=on`: cut `feature/<slug>-<n>`, stack children under it, then `merge-stack` + `ship-epic` promote to `main`. This is the behavior the L3 eval asserts as correct. **The swarm behavior change itself is split into its own OpenSpec change, `swarm-epic-arg-mode`** (it alters runbook behavior independent of evals). This `skill-evals` change's EPIC_MODE eval depends on `swarm-epic-arg-mode` landing first.
2. **L3 harness substrate → Agent SDK (Python).** The maintained behavioral suite uses the Python `query()` with `PreToolUse`/`PostToolUse` hooks for programmatic tool-call capture. (`claude -p --output-format stream-json` remains fine for ad-hoc debugging.)
3. **L4 sandbox → dedicated test-org repo.** A persistent throwaway repo in a test org, reset between runs, so the smoke tests exercise the real `gh` issue/PR/sub-issue/dependency APIs that carry the side-effect bugs. CI needs a token with repo scope to that repo only.
4. **Judge → `claude-sonnet-4-6`, moderate cap.** The LLM-as-judge grader pins `claude-sonnet-4-6` with `--max-budget-usd ~0.50` per gated/nightly L3 run. Calibrate against human labels before trusting verdicts (per the L3 requirement).
