# Ship

## Purpose

Cut a release directly from the trunk in a single step: tag HEAD of `main`, push the tag, and create a GitHub Release with autogenerated notes. This is the GitHub Flow release path — no release branch and no release PR, since `main` is always shippable.

## Requirements

### Requirement: Legacy Flow Guard

The skill SHALL refuse to run in a repository still configured for the legacy develop/RC/main flow and SHALL direct the operator to the migration helper instead.

#### Scenario: Repository on the legacy split flow

- **WHEN** the GitHub default branch is `develop`, or a remote `develop` exists while no remote `main` exists
- **THEN** the skill SHALL abort with a non-zero exit
- **AND** direct the operator to run the v4 migration helper before shipping

### Requirement: Pre-flight Release Gate

The skill SHALL refuse to ship unless the current branch is `main`, local `main` matches `origin/main` exactly, the working tree is clean, and there is at least one commit since the last release tag.

#### Scenario: Not on main

- **WHEN** the current branch is not `main`
- **THEN** the skill SHALL abort
- **AND** instruct the operator to check out `main` first

#### Scenario: Local main out of sync with origin

- **WHEN** local `main` does not match `origin/main` after fetching
- **THEN** the skill SHALL abort
- **AND** instruct the operator to sync first

#### Scenario: Dirty working tree

- **WHEN** the working tree contains uncommitted or staged changes
- **THEN** the skill SHALL abort because the release would not reflect what is committed

#### Scenario: No commits since the last tag

- **WHEN** a prior release tag exists and there are zero commits between it and HEAD
- **THEN** the skill SHALL abort reporting there is nothing to release

#### Scenario: First-ever release exemption

- **WHEN** no prior release tag exists at all
- **THEN** the progress check SHALL be skipped so the first release can proceed

### Requirement: Release Tag Derivation

The skill SHALL derive the next release tag automatically, detecting the scheme from existing tags — calver for calver repos and semver otherwise — and every tag SHALL be prefixed with `v`. Per-plugin tags containing `--v` SHALL be excluded from the release-tag search.

#### Scenario: Calver scheme detected

- **WHEN** the last release tag matches the calver shape (`vYYYY.M.D` or `vYYYY.M.D.N`), or there is no prior release tag but the newest existing release tag is calver
- **THEN** the skill SHALL derive the next calver tag from today's date as `vYYYY.M.D` with no zero-padding on month or day
- **AND** if a tag for today already exists it SHALL append `.N`, starting at `.1` and incrementing past the highest same-day suffix

#### Scenario: Calver ignores the conventional-commit bump

- **WHEN** the calver path runs
- **THEN** the major/minor/patch signal derived from conventional commits SHALL NOT affect the calver date tag

#### Scenario: Semver scheme detected

- **WHEN** no calver scheme is detected and a prior release tag exists
- **THEN** the skill SHALL bump the appropriate component of the last tag based on conventional commits since that tag
- **AND** treat any `feat` type as a minor bump, any other type as a patch bump, and any `!` after the type or `BREAKING CHANGE:` token as a major bump

#### Scenario: First semver release

- **WHEN** the scheme is semver and no prior release tag exists
- **THEN** the skill SHALL start the tag at `v0.1.0`

#### Scenario: Unknown scheme fails loudly

- **WHEN** the resolved scheme is neither calver nor semver
- **THEN** the skill SHALL abort with a non-zero exit rather than creating an empty-named tag

### Requirement: Operator Confirmation

The skill MUST present the proposed tag and its rationale and obtain explicit operator approval before creating or pushing any tag. It MUST NOT push a tag without that approval.

#### Scenario: Plan presented before mutation

- **WHEN** the tag has been derived
- **THEN** the skill SHALL display the proposed tag, the rationale, and the commits since the last tag
- **AND** prompt the operator to confirm, override, or decline

#### Scenario: Operator declines

- **WHEN** the operator declines or aborts at the confirmation prompt
- **THEN** the skill SHALL stop without creating tags, pushing, or creating a release

#### Scenario: Operator override

- **WHEN** the operator supplies an override tag
- **THEN** the skill SHALL accept it only if it matches the semver shape `vX.Y.Z` or the calver shape `vYYYY.M.D[.N]` with no pre-release suffix
- **AND** use the override tag in place of the derived one

### Requirement: Annotated Tag and Push

After confirmation, the skill SHALL create an annotated tag at HEAD of `main` and push it to origin. The tag MUST be annotated; lightweight tags SHALL NOT be used.

#### Scenario: Tag created and pushed

- **WHEN** the operator confirms the plan on a clean, in-sync `main`
- **THEN** the skill SHALL create an annotated tag carrying author and message metadata at HEAD of `main`
- **AND** push that tag to origin

### Requirement: GitHub Release Creation

The skill SHALL create a GitHub Release from the pushed tag with autogenerated notes.

#### Scenario: Release published

- **WHEN** the annotated tag has been pushed
- **THEN** the skill SHALL create a GitHub Release for that tag with autogenerated notes titled with the tag name

### Requirement: Release Report

The skill SHALL report the resulting tag, the bump type and rationale, and the release URL after creating the release.

#### Scenario: Reporting the result

- **WHEN** the GitHub Release has been created
- **THEN** the skill SHALL print the new tag, the bump type and rationale, and the release URL
